Prefer structured logging over the global `warn()`.

## Rule Details :badge[Best Practice]{variant=caution}

The global `warn()` call is fine for quick local debugging, but it doesn't scale. A logging wrapper (like `Log` from
`RbxUtil`) gives you levels, consistent output, and an easy way to filter in production.

<Aside type="tip" title="Consistent Output">
	If you care about logs later (filtering, searching, routing), use your logging wrapper instead of raw `warn()`.
</Aside>

## Refactoring Workflow

<Steps>
1. **Audit Warnings**
   Determine if the warning is still relevant or if it should be an `Error` (halting) or `Info` (informational).

2. **Pick severity** Make sure warnings are visible where you expect them (dev, staging, etc.).

3. **Convert the call** Replace `warn(...)` with `log.Warning(...)`.

4. **Verify filtering** Confirm your production build handles log filtering the way you want.

</Steps>

## Examples

<RuleExample type="fail" title="Incorrect">
		### Raw Warning Calls
		Avoid using the global `warn` for state alerts or deprecations.

    	<Code
    		code={[
    			'warn("API endpoint deprecated");',
    			'warn("Missing config property:", key);',
    			'warn("Unexpected state in component");',
    		].join("\n")}
    		lang="tsx"
    	/>
    </RuleExample>

<RuleExample type="pass" title="Correct">
    	### Structured Warnings
    	Use the warning level of your logging library for consistent telemetry.

    	<Code
    		code={[
    			'log.Warning("API endpoint deprecated");',
    			'log.Warning("Missing config property:", key);',
    			'log.Warning("Unexpected state in component");',
    		].join("\n")}
    		lang="tsx"
    	/>
    </RuleExample>

## Why use Log instead?

<CardGrid>
	<Card title="Traffic Control" icon="list-format">
		Easily suppress warnings in production while keeping them enabled in your staging builds.
	</Card>
	<Card title="Meta Data" icon="bolt">
		Attach extra context that your logger can format.
	</Card>
	<Card title="Unified Handling" icon="setting">
		Handle all application events (Info, Debug, Error) through a single, testable interface.
	</Card>
</CardGrid>
