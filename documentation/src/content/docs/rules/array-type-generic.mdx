Pick one array type style and stick with it. This rule bans the bracket syntax (`T[]`, `readonly T[]`) and rewrites it
to the generic form (`Array<T>`, `ReadonlyArray<T>`).

## Rule details :badge[TypeScript]{variant=note}

The fixer handles four cases:

1. `T[]` becomes `Array<T>`
2. `readonly T[]` becomes `ReadonlyArray<T>`
3. Nested brackets like `string[][]` become `Array<Array<string>>`
4. Tuple arrays like `[A, B][]` become `Array<[A, B]>`

Standalone tuples are left alone.

<Aside type="tip" title="Tuple types stay intact">
	`[x: number, y: number]` is valid and won't be touched. Only the array wrapper syntax gets rewritten.
</Aside>

## Examples

<RuleExample type="fail" title="Incorrect">
	<Code
		code={[
			"type Names = string[];",
			"type Values = readonly number[];",
			"type Pairs = [number, string][];",
			"type Grid = string[][];",
		].join("\n")}
		lang="tsx"
	/>
</RuleExample>

<RuleExample type="pass" title="Correct">
	<Code
		code={[
			"type Names = Array<string>;",
			"type Values = ReadonlyArray<number>;",
			"type Pairs = Array<[number, string]>;",
			"type Grid = Array<Array<string>>;",
			"type Point = [x: number, y: number];",
		].join("\n")}
		lang="tsx"
	/>
</RuleExample>

## Why bother?

<CardGrid>
	<Card title="Consistency" icon="list-format">
		One array format across the codebase means less noise in code review.
	</Card>
	<Card title="Readability at depth" icon="magnifier">
		`Array<Array<string>>` is easier to parse than `string[][]`, and `ReadonlyArray<T>` is clearer than `readonly
		T[]`.
	</Card>
	<Card title="Safe autofix" icon="bolt">
		The fixer walks the full type chain and preserves tuple semantics.
	</Card>
</CardGrid>
