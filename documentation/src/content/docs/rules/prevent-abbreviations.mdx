Prevent cryptic abbreviations in variable and property names.

## Rule Details :badge[Clean Code]{variant=success}

This rule discourages the use of abbreviations in favor of more descriptive names. It provides suggestions for replacements and can automatically fix single-replacement cases. By using full words, your code becomes more accessible to contributors and easier to maintain over time.

<Aside type="tip" title="Self-Documenting Code">
	Abbreviations like `fn` or `dist` are ambiguous. Using `function` or `distance` immediately clarifies the purpose of
	the identifier.
</Aside>

## Refactoring Steps

<Steps>
1. **Identify Abbreviations**
   The linter will flag common abbreviations like `err`, `acc`, or `idx`.

2. **Evaluate Context** Determine the full name that best describes the variable's role (e.g., `error`, `accumulator`, `index`).

3. **Check Allow-List** If an abbreviation is a widely accepted industry standard (like `props` in React), add it to your `allowList` configuration.

4. **Auto-Fix** Run `eslint --fix` to automatically convert simple abbreviations to their full-word equivalents.

{/* oxfmt-ignore */}

</Steps>

## Examples

<Tabs>
	<TabItem label="❌ Incorrect" icon="close">
		### Cryptic Names
		Bans short, ambiguous names that force the reader to guess the meaning.

    	<Code
    		code={[
    			"const err = new Error();",
    			"const fn = () => {};",
    			"const dist = calculate();",
    			"const acc = items.reduce(...);",
    		].join("\n")}
    		lang="tsx"
    	/>
    </TabItem>
    <TabItem label="✅ Correct" icon="check">
    	### Descriptive Names
    	Use complete words to make the code readable and professional.

    	<Code
    		code={[
    			"const error = new Error();",
    			"const callback = () => {};",
    			"const distance = calculate();",
    			"const accumulator = items.reduce(...);",
    		].join("\n")}
    		lang="tsx"
    	/>
    </TabItem>

</Tabs>
