Enum names should be singular, not plural.

## Rule Details :badge[Clean Code]{variant=success}

Enum types represent a single choice from a set, not the whole collection. Plural names like `Colors` make code read as
if the variable holds multiple values. This rule prefers singular names like `Color`.

<Aside type="tip" title="Natural Language">
	`const theme: Color = Color.DeepSea` reads much more naturally than `const theme: Colors = Colors.DeepSea`.
</Aside>

## Examples

<Tabs>
	<TabItem label="❌ Incorrect" icon="close">
		### Plural Names
		Bans enums named with plural suffixes or irregular plural forms.

    	<Code
    		code={[
    			"enum Colors {",
    			"\tRed,",
    			"\tBlue,",
    			"}",
    			"",
    			"// Irregular plurals",
    			"enum People {",
    			"\tAdmin,",
    			"\tUser,",
    			"}",
    		].join("\n")}
    		lang="tsx"
    	/>
    </TabItem>
    <TabItem label="✅ Correct" icon="check">
    	### Singular Names
    	Use singular forms for all enum declarations.

    	<Code
    		code={[
    			"enum Color {",
    			"\tRed,",
    			"\tBlue,",
    			"}",
    			"",
    			"enum Person {",
    			"\tAdmin,",
    			"\tUser,",
    			"}",
    		].join("\n")}
    		lang="tsx"
    	/>
    </TabItem>

</Tabs>

## Refactoring Path

<Steps>
1. **Identify Plurals**
   The linter will flag any enum that appears to be plural (including irregulars like `Feet` or `Children`).

2. **Simplify Name** Determine the singular form of the identifier.

3. **Rename & Update** Rename the enum and all references. Note that this often clarifies function arguments as well:
   `function set(color: Color)`.

4. **Exception Handling** If you have a bitmask enum where multiple flags are stored in one variable, consider adding a
   `// eslint-disable-line` with a comment explaining why plural is intended.

{/* oxfmt-ignore */}

</Steps>

## Why Singular?

<CardGrid stagger>
	<Card title="Type Clarity" icon="list-format">
		A variable of type `Color` clearly contains exactly one color.
	</Card>
	<Card title="TS Consistency" icon="bolt">
		Matches how interfaces (e.g., `User`, not `Users`) and classes are named in the ecosystem.
	</Card>
	<Card title="Better API Design" icon="rocket">
		Prevents confusing signatures like `function apply(types: Type)` which are grammatically incorrect.
	</Card>
</CardGrid>

## Detectable Plurals

The rule can spot a few common plural patterns:

- **Regular**: `Commands`, `Statuses`, `Types`.
- **Irregular**: `Children`, `People`, `Feet`, `Teeth`, `Mice`.
- **Common Suffixes**: Handles `-es`, `-ies`, and `-s` logic.
