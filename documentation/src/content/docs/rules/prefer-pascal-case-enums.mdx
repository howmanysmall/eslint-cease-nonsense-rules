Enum names and members must be PascalCase.

## Rule Details

TypeScript enums should use PascalCase for both the enum name and all members. This enforces consistency with TypeScript conventions and improves code readability.

## Options

This rule has no configuration options.

## Why PascalCase for Enums?

### 1. TypeScript Convention

PascalCase is the standard TypeScript convention for enums:

<Code
	code={[
		"// ✅ Follows TypeScript style guide",
		"enum Status {",
		"\tActive,",
		"\tInactive,",
		"}",
		"",
		"// ❌ Conflicts with conventions",
		"enum status {",
		"\tACTIVE,",
		"\tINACTIVE,",
		"}",
	].join("\n")}
	lang="typescript"
	key="follows-typescript-s-37tyo2"
/>

### 2. Consistency with Types

Enums are types, so they should follow type naming conventions:

<Code
	code={[
		"// Types use PascalCase",
		'type UserStatus = "active" | "inactive";',
		"interface UserData {",
		"\t/* ... */",
		"}",
		"class UserService {",
		"\t/* ... */",
		"}",
		"",
		"// Enums should too",
		"enum UserRole {",
		"\tAdmin,",
		"\tUser,",
		"}",
	].join("\n")}
	lang="typescript"
	key="types-use-pascalcase-2u9ubc"
/>

### 3. Distinguish from Constants

<Code
	code={[
		"// Module-level constant (SCREAMING_SNAKE_CASE)",
		"const MAX_RETRIES = 3;",
		"",
		"// Enum (PascalCase)",
		"enum RetryStrategy {",
		"\tExponential,",
		"\tLinear,",
		"\tFixed,",
		"}",
	].join("\n")}
	lang="typescript"
	key="module-level-constan-3dwqof"
/>

### 4. Better Readability

<Code
	code={[
		"// ❌ Hard to read",
		"enum HTTP_STATUS {",
		"\tOK,",
		"\tNOT_FOUND,",
		"\tINTERNAL_SERVER_ERROR,",
		"}",
		"",
		"// ✅ Easy to read",
		"enum HttpStatus {",
		"\tOk,",
		"\tNotFound,",
		"\tInternalServerError,",
		"}",
		"",
		"// Usage is clearer",
		"const status = HttpStatus.NotFound; // vs HTTP_STATUS.NOT_FOUND",
	].join("\n")}
	lang="typescript"
	key="hard-to-read-xuykb6"
/>

## Edge Cases

### Numbers in Member Names

<Code
	code={[
		"// ✅ Numbers are allowed",
		"enum Version {",
		"\tV1,",
		"\tV2,",
		"\tV3,",
		"}",
		"",
		"enum Level {",
		"\tLevel1,",
		"\tLevel2,",
		"\tLevel3,",
		"}",
	].join("\n")}
	lang="typescript"
	key="numbers-are-allowed-1lm58v"
/>

### String Literal Members

<Code
	code={[
		"// ✅ String literal keys must also be PascalCase",
		"enum Status {",
		'\t"Active" = "ACTIVE", // ✅ Member name is PascalCase',
		'\t"Inactive" = "INACTIVE", // ✅ Good',
		"}",
		"",
		"// ❌ Bad",
		"enum Status {",
		'\t"active" = "ACTIVE", // Member name not PascalCase',
		'\t"INACTIVE" = "inactive", // Member name SCREAMING_SNAKE_CASE',
		"}",
	].join("\n")}
	lang="typescript"
	key="string-literal-keys--hsuzah"
/>

### Numbers Starting Member Names

Members starting with digits are skipped:

<Code
	code={[
		"// Numbers at start - not checked by this rule",
		"enum Code {",
		'\t"200" = "OK",',
		'\t"404" = "Not Found",',
		"}",
	].join("\n")}
	lang="typescript"
	key="numbers-at-start-not-157jk1"
/>

## When Not To Use It

Disable this rule if:

- Your project uses different enum naming conventions
- You're working with generated code that uses other conventions
- You're integrating with external libraries that expect specific naming

		"}",
		"",
		"// ✅ Consistent",
